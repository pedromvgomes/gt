## Highlights

- **Auto-update.** `gt` now checks GitHub Releases at most once per day and offers
  to install newer versions in place. Run `gt update` to update on demand, or
  `gt update --check` to only report availability.
- **Setup templates.** New `gt setup` and `gt config` commands let you declare
  per-repo bootstrap steps in `~/.config/gt/config.yaml` and run them on demand.

## Auto-update

After a successful command, `gt` may print a one-liner like:

```
INFO: gt v0.3.0 is available (you're on v0.2.0). Run 'gt update' to install.
```

The check is skipped automatically when:

- the binary was built from source (`version == dev`),
- `GT_NO_UPDATE_CHECK=1` or `CI` is set,
- stdout/stderr is not a TTY,
- the binary lives under a package-manager prefix (Homebrew, Nix).

Disable globally by editing your config:

```yaml
auto_update:
  enabled: false
```

Or change the cadence (minimum `1h`, default `24h`):

```yaml
auto_update:
  check_interval: 12h
```

`gt update` verifies the release tarball against `checksums.txt`, replaces the
running binary atomically, and preserves its file mode.

## Setup templates

`gt setup` runs configured templates against the current repository; templates
are matched by repo URL globs and each declares either an inline `run:` block or
a `script:` path. See `gt config show` for the seeded YAML and inline docs.

`gt config path | show | validate | edit` round out config management.
